# Csrf

## Overview
CSRF (Cross-Site Request Forgery) is a type of cyber attack where an attacker tricks a user into performing unintended actions on a web application that the user is authenticated to. This concept matters because it can lead to unauthorized data modification, sensitive information disclosure, or even financial loss. Understanding CSRF is crucial for web developers to protect their applications and users from such attacks.

## Key Concepts
- **CSRF Attack**: An attack where an attacker tricks a user into performing unintended actions on a web application.
- **CSRF Token**: A unique token generated by the server to validate requests and prevent CSRF attacks.
- **Same-Origin Policy**: A security feature implemented in web browsers to prevent web pages from making requests to a different origin (domain, protocol, or port) than the one the web page was loaded from.

## Detailed Explanation
CSRF attacks typically occur when an attacker creates a malicious web page that makes a request to a vulnerable web application. The request is made in such a way that it appears to come from the user's browser, and the web application, trusting the request, performs the action. To prevent such attacks, web applications can use a CSRF token, which is a unique token generated by the server and included in every request. The server then verifies the token on each request to ensure that it is valid and has not been tampered with.

The same-origin policy is another important concept in preventing CSRF attacks. This policy restricts web pages from making requests to a different origin than the one the web page was loaded from. However, attackers can bypass this policy by using techniques such as JSONP (JSON with Padding) or CORS (Cross-Origin Resource Sharing). To counter this, web applications can implement additional security measures, such as validating the origin of requests and using secure protocols for communication.

CSRF attacks can be categorized into three types: GET-based, POST-based, and JSON-based. GET-based attacks occur when an attacker makes a GET request to a vulnerable web application, while POST-based attacks occur when an attacker makes a POST request. JSON-based attacks are similar to POST-based attacks but use JSON data instead of form data. To prevent these attacks, web applications can use a combination of CSRF tokens, same-origin policy, and input validation.

In addition to these measures, web applications can also use other techniques to prevent CSRF attacks, such as using a double-submit cookie or a header-based token. A double-submit cookie is a cookie that is set by the server and included in every request. The server then verifies the cookie on each request to ensure that it is valid and has not been tampered with. A header-based token is similar to a CSRF token but is included in the request headers instead of the request body.

## Code Examples

### Example 1: Basic Usage
```python
from flask import Flask, request, session
from flask_wtf import FlaskForm
from wtforms import StringField
from wtforms.validators import DataRequired

app = Flask(__name__)
app.config['SECRET_KEY'] = 'secret_key'

class MyForm(FlaskForm):
    name = StringField('name', validators=[DataRequired()])

@app.route('/submit', methods=['POST'])
def submit():
    form = MyForm()
    if form.validate_on_submit():
        # Validate CSRF token
        if request.form['csrf_token'] == session['csrf_token']:
            # Perform action
            return 'Action performed successfully'
        else:
            return 'Invalid CSRF token'
    else:
        return 'Form validation failed'

@app.route('/form', methods=['GET'])
def form():
    # Generate CSRF token
    session['csrf_token'] = 'csrf_token'
    return '''
    <form action="/submit" method="post">
        <input type="hidden" name="csrf_token" value="{}">
        <input type="text" name="name">
        <input type="submit" value="Submit">
    </form>
    '''.format(session['csrf_token'])
```
**Explanation:** This code example demonstrates a basic CSRF protection mechanism using Flask and Flask-WTF. The `submit` function validates the CSRF token on each request, and the `form` function generates a new CSRF token on each GET request.

### Example 3: Advanced Pattern
```python
from flask import Flask, request, session
from flask_wtf import FlaskForm
from wtforms import StringField
from wtforms.validators import DataRequired
from flask_wtf.csrf import CSRFProtect
from flask_login import LoginManager, UserMixin, login_required

app = Flask(__name__)
app.config['SECRET_KEY'] = 'secret_key'
csrf = CSRFProtect(app)
login_manager = LoginManager(app)

class User(UserMixin):
    def __init__(self, id):
        self.id = id

@login_manager.user_loader
def load_user(user_id):
    return User(user_id)

class MyForm(FlaskForm):

@app.route('/submit', methods=['POST'])
@login_required

    form = MyForm()

        <input type="text" name="name">

    </form>
    '''
```
**Explanation:** This code example demonstrates an advanced pattern of CSRF protection using Flask, Flask-WTF, Flask-WTF-CSRF, and Flask-Login. The `login_required` decorator ensures that only authenticated users can access the form and submit actions.

## Common Mistakes
1. **Not validating CSRF tokens**: Failing to validate CSRF tokens on each request can leave the application vulnerable to CSRF attacks.
2. **Not generating new CSRF tokens**: Failing to generate new CSRF tokens on each request can allow attackers to reuse tokens and perform unauthorized actions.
3. **Not using secure protocols**: Failing to use secure protocols such as HTTPS can allow attackers to intercept and modify requests, including CSRF tokens.

## Best Practices
- **Use a well-established CSRF protection library**: Using a well-established library such as Flask-WTF-CSRF can simplify the process of implementing CSRF protection.
- **Generate new CSRF tokens on each request**: Generating new CSRF tokens on each request can prevent attackers from reusing tokens.
- **Validate CSRF tokens on each request**: Validating CSRF tokens on each request can prevent attackers from performing unauthorized actions.
